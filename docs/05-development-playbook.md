## 2. 开发总原则
1. **默认进入 Plan 模式，而不是立即开发**
2. **Plan 阶段允许和用户交流想法，不做副作用操作**
3. **只有在计划确认后，才进入 Execute 阶段**
4. **实现必须遵循 TDD：先测后码**
5. **代码变更后必须同步修正文档**
6. **高风险操作必须可见、可控、可回退**

---

## 3. Plan 模式如何工作
Plan 模式是 CodePilot 的第一入口，也是最重要的用户体验设计之一。

### 3.1 Plan 模式目标
Plan 模式要做的不是“给一个简短建议”，而是形成可讨论、可确认、可执行的开发草案。

### 3.2 Plan 模式必须输出的内容
每次进入 Plan 模式，系统至少输出：
1. **任务摘要**：用户要做什么
2. **假设与未知项**：哪些条件未确认
3. **候选文件**：可能需要读取/修改的文件
4. **候选命令**：可能会运行的测试、构建、检查命令
5. **步骤计划**：按顺序列出的执行步骤
6. **风险点**：例如删除、覆盖配置、运行数据库迁移
7. **下一步选择**：
   - 继续讨论计划
   - 调整计划
   - 确认执行
   - 取消任务

在 CLI 中，建议把这些状态映射为显式命令：
- 输入自然语言或 `/plan <需求>`：进入待确认计划
- `/approve`：执行当前计划
- `/cancel`：放弃当前计划
- `/status`：查看当前待确认计划、命令边界和工作目录

### 3.3 Plan 模式期间禁止的事情
在未确认前，禁止：
- 修改生产代码
- 写入关键配置
- 执行危险命令
- 启动具有副作用的部署流程

允许的事情：
- 读文件
- 搜索代码
- 扫描目录
- 生成计划
- 分析风险

结论：Plan 模式是**讨论与设计阶段**，不是隐性执行阶段。

## 4. Execute 模式如何启动
只有同时满足以下条件，才进入 Execute：
- 计划已明确
- 用户已确认
- 风险项已接受或已排除
- 执行边界已经确定（目录、命令、文件范围）

执行过程中要遵循：
1. 先读再改
2. 先局部编辑，后整文件重写
3. 每次关键变更后立即做语法检查
4. 每轮修改后更新测试结果和 Diff

## 5. 按模块实施的开发顺序

## 5.1 第一优先级：Planner 与交互框架
先做：
- 任务请求模型
- Plan/Execute 状态机
- 计划输出结构
- 用户确认接口

原因：如果没有 Plan 框架，后续所有工具能力都缺少统一入口。

### 交付标准
- 能区分 `plan` 与 `auto`
- `plan` 模式能停在等待确认
- 计划输出包含步骤、文件、命令和风险项

---

## 5.2 第二优先级：搜索与读取能力
先做：
- Glob 搜索
- Grep 搜索
- 带行号读文件
- 大文件范围读取

原因：Agent 想改代码，必须先准确找到代码。

### 交付标准
- 能找测试文件、配置文件、入口文件
- 能通过 grep 找函数/类/错误字符串
- 能按行读取代码片段并保留实际行号
- 在交互式 shell 中可直接通过 `/files`、`/grep`、`/read`、`/replace` 调用这些能力

---

## 5.3 第三优先级：编辑能力
先做：
- old string → new string 替换
- 修改失败时明确报错
- 修改后输出 diff
- 修改后自动语法检查

后做：
- 首尾锚点匹配
- 更复杂的多段编辑策略

原因：过早追求复杂编辑器，会在课程周期里显著拉高风险。

---

## 5.4 第四优先级：命令执行与环境保持
先做：
- 持久 shell 会话
- 指定工作目录执行
- 退出码、stdout、stderr 捕获
- 输出头尾截断

后做：
- 独立并行终端
- 后台进程管理
- 更细粒度的环境快照

原因：测试与验证是 Coding Agent 可信度的关键。

---

## 5.5 第五优先级：Code Interpreter
先做：
- 安全执行小段 Python
- 超时控制
- 输出截断

后做：
- 更强沙盒
- 资源隔离
- 工件管理

原因：它很重要，但在课程原型里优先级略低于搜索、编辑和测试执行主链路。

---

## 6. TDD 落地要求
本项目所有新增核心能力，都应按以下顺序开发：
1. 先写失败测试
2. 运行测试确认失败原因正确
3. 写最小实现
4. 再运行测试确认通过
5. 运行完整测试防止回归
6. 同步更新相关文档

### 推荐顺序
- 先写单元测试
- 再写黑盒场景测试
- 最后补白盒覆盖与异常路径

### 不允许的做法
- 先写完一堆代码，最后补测试
- 文档先写成完整功能，但代码没有对应骨架
- 修改行为后不更新测试与文档

## 7. 文档同步规则
开发中必须遵守以下同步关系：

### 设计变化时
若修改了：
- 工具能力定义
- Plan/Execute 状态
- 搜索或编辑策略
- 执行边界规则

必须同步更新：
- `docs/02-technical-route.md`
- 若影响主流程，也要更新 `docs/01-project-development.md`

### 迭代安排变化时
必须同步更新：
- `docs/03-iteration-plan.md`

### 测试策略变化时
必须同步更新：
- `docs/04-testing-strategy.md`

结论：**文档不是收尾材料，而是开发中的约束文件。**

## 8. 推荐的实现里程碑

### 里程碑 A：Plan 模式可用
验收条件：
- 能解析任务
- 能输出计划
- 能展示风险
- 能停在等待确认

### 里程碑 B：搜索与读取可用
验收条件：
- Glob / Grep 可用
- 带行号文件读取可用
- 大文件范围读取可用

### 里程碑 C：编辑与语法反馈可用
验收条件：
- 能对单文件做局部修改
- 修改后立即返回 diff
- 修改后立即返回语法检查结果

### 里程碑 D：执行与验证可用
验收条件：
- 持久 shell 可用
- 能运行测试/lint/build
- 长输出可截断并存档

### 里程碑 E：Agent 闭环可演示
验收条件：
- 从计划到修改到验证形成闭环
- 失败时能给出修复或回退建议

### 里程碑 F：Sprint 2 增强能力可运行
验收条件：
- CLI 可直接运行 `codepilot run/history/restore`
- 交互式 shell 支持 `/plan`、`/approve`、`/status`、`/files`、`/grep`、`/read`、`/replace`
- `codepilot --tui` 可进入阶段 B 的最小多面板 TUI 骨架，并在非 TTY 环境输出可测试快照
- 当前 TUI 骨架至少支持左侧 session / log 列表切换，以及右侧 detail 跟随当前选中项展示 summary / context
- 右侧 detail 仍支持 diff / planner trace / failure hints，并可对较长内容做基础滚动/分页浏览
- `.env` 中配置的 DeepSeek Key 可驱动真实计划生成
- 会话历史、日志与回退快照可在本地存储中追踪
- 命令边界与回退路径有实际测试证明

## 9. 按本文档开发时的推荐操作顺序
每接到一个新需求，建议按以下顺序推进：
1. 查阅旧文档确认该需求落在哪个模块
2. 先进入 Plan 模式，形成计划并与用户讨论
3. 把计划落到具体文件和测试点
4. 先写失败测试
5. 做最小实现
6. 运行静态检查与测试
7. 更新文档
8. 再进入下一轮实现

这意味着 CodePilot 的开发本身，也要遵循 CodePilot 所提倡的工作方式。

---

## 10. 最终结论
本文档的意义在于把旧文档串成一套真正能执行的开发流程：
- 旧文档回答“系统是什么、为什么这么设计”
- 本文档回答“接下来怎么一步一步做出来”

最终开发应遵循以下一句话：

> **先按旧文档理解系统，再按本文档进入 Plan 模式讨论方案，确认后用 TDD 实施，并在每次实现后同步修正文档。**
