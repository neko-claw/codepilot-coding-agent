# 技术路线选择

## 1. 技术路线选择框架
在本项目中，技术路线选择遵循四个标准：
1. **易实现**：适合课程周期内完成
2. **易验证**：便于展示和测试
3. **可扩展**：后续可演进为完整 Agent
4. **可维护**：模块职责清晰，避免早期过度耦合

## 2. 候选方案对比

### 2.1 实现语言
| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| Python | 生态完整，适合 Agent、测试、命令执行、脚本控制 | 前端能力弱于 JS 全栈 | 选用 |
| Java | 工程化强 | 原型迭代速度慢 | 不选 |
| Node.js | Web 集成方便 | 系统脚本与静态分析整合不如 Python 顺手 | 作为前端备选 |

**选择结果：** 后端原型与 Agent 核心使用 Python。

### 2.2 前端/原型交互方案
| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| Web 原型（React/Next.js） | 展示效果好，适合课程答辩 | 开发量相对较大 | 可作为后续演进 |
| CLI/TUI 原型 | 实现成本低，便于集成命令执行 | 视觉表现一般 | 原型阶段优先 |
| 桌面端（Electron/Tauri） | 本地能力强 | 技术栈复杂 | 本阶段不优先 |

**选择结果：** 原型阶段优先采用 Python 核心 + 可扩展 UI 接口，界面可先用低成本方案表达主流程。

### 2.3 代码编辑能力实现方式
| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| 直接整文件覆盖 | 实现简单 | 易引入误改 | 仅用于原型早期 |
| Patch/Diff 驱动修改 | 可解释性强，便于审查和回退 | 设计稍复杂 | 推荐 |
| AST 级编辑 | 精确性高 | 开发成本高 | 后续增强 |

**选择结果：** 采用“结构化计划 + Patch/Diff 展示”的稳健路线。

### 2.4 命令执行与验证
| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| 直接 shell 执行 | 开发快 | 风险高 | 必须配合白名单 |
| 容器隔离执行 | 安全性好 | 环境配置复杂 | 后续增强 |
| 沙箱执行 | 最安全 | 成本高 | 本课程阶段不优先 |

**选择结果：** 先采用受控 shell 执行，配合工作目录限制、命令白名单、危险操作确认。

## 3. 总体架构设计
```text
用户输入
   ↓
任务交互层
   ↓
仓库理解层 ──→ 上下文摘要
   ↓
任务规划层 ──→ 执行计划 + 风险标签
   ↓
代码编辑层 ──→ 变更结果 + Diff
   ↓
命令执行层 ──→ 构建/测试日志
   ↓
验证回退层 ──→ 结果确认 / 回滚
   ↓
历史记录层
```

## 4. 核心技术选型
### 4.1 Python 核心框架
- `dataclasses`：定义任务对象与风险评估对象
- `pathlib`：统一处理路径
- `subprocess`：执行 shell 命令
- `pytest`：测试主框架
- `pytest-cov`：覆盖率统计
- `ruff`：风格与常见错误检查
- `pylint`：静态质量分析

### 4.2 质量保障路线
- **Ruff**：快速检查导入、格式、未使用变量、常见错误
- **Pylint**：补充设计质量、可读性与复杂度问题
- **Pytest**：覆盖黑盒、白盒、单元和集成测试
- **Coverage**：统计语句覆盖与分支覆盖

## 5. 为什么不优先选择更复杂方案
### 不优先上完整 IDE 插件
因为课程阶段更需要证明核心执行闭环，而不是投入大量时间做编辑器适配。

### 不优先做多仓库、多代理调度
因为这会显著提高系统复杂度，不利于在 8 周内完成可验证交付。

### 不优先做全量 AST 编辑器
因为当前实验更看重“功能完整性 + 测试与原型说明”，而不是工业级编辑精度。

## 6. 技术路线最终结论
本项目采用以下技术路线：
1. **Python 作为 Agent 核心实现语言**
2. **以模块化架构组织任务输入、规划、执行、验证与回退能力**
3. **以 Diff + 日志 + 风险确认保障可解释性与安全性**
4. **以 Ruff + Pylint + Pytest + Coverage 建立基础质量门禁**
5. **以两轮迭代完成“核心闭环 → 增强能力”的开发路径**

这一方案在课程约束下更稳健：实现成本可控，展示路径清晰，测试落地容易，后续也能继续演进为更完整的 Coding Agent。
