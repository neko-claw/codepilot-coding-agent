# 技术路线选择

## 1. 技术路线目标
CodePilot 不是“能回答代码问题的聊天机器人”，而是一个真正执行开发任务的 Coding Agent。它的技术路线必须同时满足四个目标：
1. **可执行**：能读代码、搜代码、改代码、跑命令、做验证
2. **可控**：默认先规划，再执行；高风险动作必须被拦截或确认
3. **高效**：尽量并行、流式、按需读取，避免浪费时间和上下文
4. **可验证**：所有关键路径都能通过测试、日志、Diff 和环境信息证明

因此，本项目不采用“单轮对话 + 一把梭执行”的粗放模式，而采用：
- **Plan-Execute 交互模型**
- **工具增强型 Agent 架构**
- **TDD 驱动的实现方式**
- **动态上下文与持久执行环境**

---

## 2. Coding Agent 必备能力
本项目把以下 7 类能力视为基础能力，不是附属功能。

### 2.1 核心工具能力矩阵
| 能力 | 主要用途 | 实现要求 | 风险控制 |
|---|---|---|---|
| Code Interpreter | 安全执行 Python 片段，做数据处理、格式转换、验证脚本 | 隔离 Python 进程或沙盒；超时；输出上限 | 禁网、限时、限制 CPU/内存 |
| Bash Shell | 执行测试、构建、lint、脚本与文件处理命令 | 持久终端会话 + 可选隔离会话；支持 stdout/stderr、退出码 | 命令白名单、工作目录限制、危险命令确认 |
| 读文件工具 | 读取代码、配置、日志、文档 | 支持行号、范围读取、头尾截断 | 路径边界控制、读取长度限制 |
| 写文件工具 | 新建文件或整文件重写 | 原子写入、失败回滚、写后检查 | 禁止越权路径、关键文件覆盖确认 |
| 编辑文件工具 | 对现有文件做局部修改 | 支持精确定位与失败回退 | 修改后自动 diff / 语法检查 |
| Glob 搜索 | 按路径模式找文件 | 支持递归模式、文件类型过滤 | 限制根目录与返回数量 |
| Grep 搜索 | 按内容模式找文本 | 支持正则、上下文、文件过滤 | 限制返回条数和片段长度 |

### 2.2 扩展但重要的搜索能力
除了上面 7 类基础能力，真正高质量的 Coding Agent 还应逐步支持以下两类高级搜索：

| 能力 | 用途 | 技术路线 |
|---|---|---|
| 语义代码搜索 | 在陌生代码库中找“与某功能相关”的实现 | AST 分块 + 向量检索 + BM25 + reranker |
| 符号级定义/引用查找 | 精确追踪函数、类、方法的定义和调用点 | LSP / 语言分析器 / 索引数据库 |

结论上，本项目将搜索体系分为四层：
1. **Glob**：先找文件
2. **Grep**：再找文本
3. **Semantic Search**：探索语义相关代码
4. **Symbol Search**：追踪结构化定义与引用

这四类搜索形成互补工具箱，而不是彼此替代。

## 3. 总体架构设计

### 3.1 分层架构
```text
用户输入
   ↓
交互层（CLI / Web / Chat）
   ↓
Plan 控制层
   ↓
上下文汇聚层
   ├── 代码搜索
   ├── 文件读取
   ├── 环境提示注入
   └── 历史轨迹压缩
   ↓
Execute 调度层
   ├── Code Interpreter
   ├── Bash Shell
   ├── Read / Write / Edit File
   ├── Glob Search
   └── Grep Search
   ↓
验证反馈层
   ├── Diff
   ├── 语法检查
   ├── 测试结果
   └── 回退建议
   ↓
日志与历史层
```

### 3.2 模块职责
| 模块 | 职责 |
|---|---|
| `core/` | 请求模型、响应模型、任务状态、会话状态 |
| `planner/` | Plan 模式、执行计划生成、用户确认状态机 |
| `tools/` | 各类工具能力定义、参数模式、返回格式 |
| `executor/` | 工具调度、并行执行、流式结果聚合 |
| `safety/` | 风险识别、路径边界、命令拦截、确认门禁 |
| `storage/` | 快照、日志、历史任务、临时输出存档 |
| `ui/` | 展示计划、日志、Diff、风险提示与确认入口 |

## 4. 为什么必须采用 Plan-Execute 模式

### 4.1 根本原因
Coding Agent 一旦直接执行，就会立刻遇到三个工程问题：
1. 用户无法预先知道将修改哪些文件
2. 用户无法知道将运行哪些命令
3. 高风险操作没有人工确认窗口

因此，本项目的默认行为必须是：**先计划，再执行**。

### 4.2 Plan 阶段的职责
Plan 模式只负责：
- 理解需求
- 搜索相关模块
- 读取必要代码片段
- 生成步骤化计划
- 标注将调用的工具
- 列出拟修改文件、拟执行命令、风险提示
- 停在“等待确认”状态

### 4.3 Execute 阶段的职责
只有当以下条件满足时才进入执行：
- 用户确认计划
- 风险命令已接受
- 执行边界合法

执行阶段负责：
- 调用工具做文件修改
- 执行测试 / lint / 构建
- 汇总日志与 Diff
- 在失败时继续迭代修复或给出回退建议

### 4.4 Plan 模式的人机交互要求
Plan 模式必须允许用户交流想法，而不是直接进入开发。至少要支持：
- 展示任务摘要
- 展示步骤列表
- 展示目标文件和目标命令
- 展示风险标签
- 支持“继续讨论计划”“确认执行”“取消任务”

这意味着 Plan 模式不是一个静态页面，而是一个**可迭代协商的阶段**。

## 5. 高效执行技巧

## 5.1 并行工具调用与流式执行
传统串行 Agent 的流程是：模型输出一个工具调用 → 执行 → 返回 → 再生成下一个调用。这样做简单，但效率低。

更合理的做法是：
1. 当模型一次生成多个独立工具调用时，允许并行执行
2. 当解析器在流式输出中识别出一个完整工具调用时，立即启动该工具，而不是等待模型整段输出完成
3. 将多个工具结果聚合后再返回给模型，减少往返轮数

适用场景包括：
- 同时 grep 多个模式
- 同时读取多个小文件
- 同时检查配置、测试入口和 README
- 并行执行 BM25 与向量检索，再统一 rerank

### 并行执行约束
并行并不意味着毫无限制。应遵循：
- 同一文件的编辑操作不得并行
- 同一持久终端中的交互式命令不得并行
- 对共享状态有副作用的命令必须串行
- 搜索、读取、无副作用分析类工具优先并行

## 6. 上下文的精细化管理
Coding Agent 面对的是大型代码库，而模型上下文始终有限。因此上下文必须按需管理，而不是整库灌入。

### 6.1 按范围读取文件
读文件工具必须支持：
- 整文件读取（小文件）
- 按行号范围读取（大文件）
- 带行号返回

推荐返回格式：
```text
42|def create_user(...):
43|    ...
44|    return result
```

这样模型就能精确引用“第 42 行附近”，降低编辑歧义。

### 6.2 大输出的头尾截断
终端输出不能无脑注入上下文。对于超长输出，应使用：
- 保留头部若干行
- 保留尾部若干行
- 中间替换为省略提示
- 完整输出写入日志文件以供按需查看

推荐格式：
```text
[output truncated: showing first 60 and last 60 lines, full log saved to ...]
```

### 6.3 历史上下文压缩
长链任务需要保留：
- 当前计划
- 最近一次工具结果摘要
- 未解决错误
- 当前文件状态

而不需要反复保留所有历史原文。应使用摘要与结构化状态对象，而不是全文拼接。

## 7. 环境信息的动态注入
Coding Agent 严重依赖当前环境。每次推理前，应动态注入以下环境信息作为 System Hint：
- 当前工作目录
- 当前 git 分支
- 最近若干条 git 提交
- staged / unstaged diff 摘要
- 当前激活虚拟环境（若有）
- 当前运行中的关键后台进程（若相关）

### 为什么不能写死在系统提示词里
如果把这些信息硬编码进静态系统提示，会降低缓存效率，并且信息很快过时。更好的做法是：
- 保持静态系统提示稳定
- 将实时环境信息作为每轮推理前追加的动态 hint 注入

这样可以兼顾：
- KV cache 复用效率
- 环境信息的新鲜度
- 上下文长度可控

---

## 8. 命令执行环境的状态持久化
许多任务依赖 shell 状态，例如：
- 切换目录
- 激活虚拟环境
- 设置环境变量
- 启动测试服务

如果每次都在全新 shell 执行命令，这些状态会全部丢失，Agent 会不断重复样板命令。

### 8.1 推荐方案
系统应维护一个**持久终端会话**，在整个任务期间共享：
- 当前目录
- 环境变量
- 虚拟环境
- 后台服务状态

### 8.2 何时新建隔离会话
以下情况应新建独立终端：
- 需要并行执行相互独立的任务
- 需要隔离不同依赖环境
- 需要避免污染主会话状态
- 需要运行高风险或实验性命令

结论：默认使用持久会话；有隔离需求时再开新会话。

## 9. 即时语法反馈机制
文件写入后不应等待用户再要求检查。更高效的方式是：
- 每次写入或编辑后自动触发相应语法检查
- 将检查结果作为工具返回的一部分
- 若出现语法错误，让 Agent 立即在下一轮修复

### 9.1 最小策略
- Python 文件：`python -m py_compile` 或 Ruff / Pylint 相关检查
- JS/TS 文件：`tsc --noEmit`、ESLint 或框架自带检查
- JSON/YAML：基础解析校验

### 9.2 价值
这样可以把“引入错误”与“修复错误”压缩到同一轮上下文附近，减少后续排障成本。

## 10. 搜索工具体系

### 10.1 正则表达式内容搜索（Grep）
适用于：
- 已知函数名
- 已知变量名
- 已知错误消息
- 已知注释模式

优点：快、准、成本低。  
局限：不理解语义。

### 10.2 文件名模式匹配（Glob）
适用于：
- 找测试文件
- 找配置文件
- 找模块目录
- 找特定后缀文件

优点：速度快、结构清晰。  
局限：只能找路径，不能找内容。

### 10.3 语义代码搜索
适用于：
- 在陌生代码库中找“与某功能相关”的实现
- 根据自然语言找相似逻辑

技术路线：
- AST 感知分块
- 向量索引
- BM25 关键词并行召回
- reranker 统一排序

### 10.4 符号级定义与引用查找
适用于：
- 精确重命名
- 查找所有真实调用点
- 跨模块追踪引用关系

### 10.5 搜索策略组合
标准组合顺序通常是：
1. 用 Glob 找候选文件
2. 用 Grep 找具体实现
3. 用 Semantic Search 扩展未知区域
4. 用 Symbol Search 做最终结构追踪

## 11. 文件编辑工具的设计取舍
文件编辑是 Agent 最脆弱的部分之一，需要明确权衡。

### 11.1 Diff String + Apply Diff Model
优点：表达力强。  
缺点：需要第二个模型解释 diff，容易歧义，不稳健。

### 11.2 Old String → New String
优点：实现简单、失败明确。  
缺点：替换大段代码时要复制很多原文，token 浪费明显。

### 11.3 行号定位编辑
优点：删除大段内容高效。  
缺点：模型容易数错行号；编辑后行号会漂移。

### 11.4 类 Vim 编辑命令
优点：重排代码效率高。  
缺点：命令语义复杂，小模型容易出错。

### 11.5 首尾锚点匹配
形式为：
- old string start
- old string end
- new string

优点：兼顾定位稳健性与长段替换效率。  
缺点：需要框架额外实现锚点匹配逻辑。

### 11.6 本项目建议路线
课程原型阶段，建议采用分层策略：
1. **首选 old string → new string**，因为实现最稳健
2. 大段修改时，扩展为**首尾锚点匹配**
3. 暂不采用需要二次模型解释的 apply-diff 架构
4. 所有编辑后必须立即输出 diff 并做语法检查

## 12. Agent 的自举与范例优先开发
从零生成一个新的 Agent，错误率往往高于“复制一个正确范例再做局部修改”。这在 Agent 开发中尤其明显。

### 12.1 为什么直接从零生成容易出问题
常见问题包括：
- 上下文格式不规范
- 工具定义过于简陋
- API 选型过时
- 搜索、编辑、执行工具之间没有边界
- 忽略缓存、并行、状态持久化等工程问题

### 12.2 更稳健的路线
开发新 Agent 时，应优先：
1. 复制一个经过验证的 Agent 框架
2. 保留其上下文管理、工具协议、执行框架
3. 只对角色、工具集、业务逻辑做局部调整
4. 在修改基础上继续做 TDD 和文档同步

### 12.3 对 CodePilot 的启示
CodePilot 的开发文档必须明确：
- 不是从零散思路拼装
- 而是在稳定架构上逐步扩展能力
- 文档、代码、测试必须同步演进

---

## 13. 开发方法：TDD + 文档同步
本项目继续坚持 TDD：
1. 先写失败测试
2. 再写最小实现
3. 验证通过后重构
4. 最后同步更新文档

同时加入一条额外约束：
- **设计变更必须同步修正文档**
- **文档新增约束必须落实到代码或测试骨架**

这能避免“文档是理想状态，代码是另一套系统”的脱节。

## 14. 与课程实验交付的关系
这套技术路线能够直接支撑实验要求：
- **项目概述**：明确 Coding Agent 与普通聊天系统的差别
- **用户故事**：围绕 plan、搜索、编辑、执行、验证闭环展开
- **原型设计**：Plan 页、执行页、Diff 页、日志页都有明确依据
- **两轮迭代**：第一轮实现工具与主流程；第二轮强化持久会话、日志、回退与高级搜索
- **测试方案**：可从 TDD、黑盒、白盒、覆盖率多层验证

---

## 15. 最终结论
本项目最终采用以下技术路线：
1. **以 Python 作为 Coding Agent 核心实现语言**
2. **以 7 类基础工具能力构建执行闭环，并预留语义搜索与符号搜索扩展位**
3. **以 Plan-Execute 模式作为默认交互主流程，支持在 Plan 阶段与用户讨论而不执行开发**
4. **以并行工具调用、流式执行、动态上下文、持久终端来提升效率**
5. **以按范围读取、头尾截断、环境动态注入来控制上下文成本**
6. **以即时语法反馈、Diff、日志与回退来保证可靠性**
7. **以 TDD 和文档同步作为开发纪律**
8. **以成熟范例为基础进行修改，而不是从零任意拼装 Agent**

这一路线比单纯的“模型 + 工具”描述更接近真正的工程实现：它明确了性能、上下文、风险、交互和演化路径，因而可以作为后续开发的基准文档。
