# 技术路线选择

## 1. 技术路线选择原则
本项目不是一般的聊天机器人，而是一个需要真正“看代码、做计划、改文件、跑命令、验证结果”的 Coding Agent。因此技术路线必须围绕**可执行性、安全性、可解释性、可验证性**展开，而不是停留在“接一个大模型 API”这一层。

本项目采用以下判断标准：
1. **能力完整**：必须覆盖 Coding Agent 的核心工具链
2. **过程可控**：先规划、后执行，避免直接做高风险修改
3. **实现可落地**：适合课程周期与团队能力
4. **测试可验证**：每个关键能力都能被静态检查与自动化测试覆盖
5. **后续可扩展**：可逐步演进为更完整的 Agent 平台

---

## 2. Coding Agent 必备能力与技术实现
用户已明确要求以下 7 项能力，这些能力不是可选增强，而是系统设计的基础约束。

### 2.1 工具能力矩阵
| 能力 | 作用 | 技术实现路线 | 风险控制 |
|---|---|---|---|
| Code Interpreter | 在隔离环境中执行 Python 代码，用于数据处理、代码验证、快速脚本执行 | Python 沙盒执行器；独立进程/受限环境；超时与输出限制 | 限制网络、CPU/内存、执行时长 |
| Bash Shell | 执行测试、构建、格式化、文件处理等命令 | 受控 `subprocess` 或容器 shell | 命令白名单、工作目录限制、危险命令确认 |
| 读文件工具 | 读取代码、配置、日志、文档 | 基于 `pathlib` / 文件系统 API | 只读目录控制、最大读取长度 |
| 写文件工具 | 新建文件或整体重写文件 | 原子写入、备份快照 | 禁止越权路径、关键文件覆盖确认 |
| 编辑文件工具 | 局部修改文件，减少整文件误改 | patch / diff / 定位替换 | 修改前后 diff 校验 |
| Glob 搜索 | 快速匹配文件名，如 `*.py`、`**/test*.py` | `glob` / `pathlib.rglob()` | 限制扫描根目录 |
| Grep 搜索 | 在代码和文档中查找关键文本模式 | `re` / ripgrep 思路封装 | 限制返回条数与上下文长度 |

### 2.2 为什么这 7 项能力缺一不可
1. **没有 Code Interpreter**：Agent 无法安全执行 Python 片段，很多数据转换、验证和实验性推理无法落地。
2. **没有 Bash Shell**：Agent 不能运行测试、构建与脚本，最终只能“写代码但不能证明代码可运行”。
3. **没有读写编辑工具**：Agent 无法形成对文件系统的闭环操作。
4. **没有 Glob/Grep**：Agent 难以高效定位项目入口、配置、测试文件与错误位置。

因此本项目不将 Agent 定义为“会回答代码问题的模型”，而定义为“围绕仓库执行任务的工具增强型系统”。

## 3. 总体架构设计

### 3.1 分层架构
```text
用户请求
   ↓
交互层（UI / CLI / Web）
   ↓
Plan 模式控制层
   ↓
任务规划层（分析目标、拆解步骤、识别风险、列出将调用的工具）
   ↓
Execute 模式调度层
   ├── Code Interpreter
   ├── Bash Shell
   ├── Read / Write / Edit File
   ├── Glob Search
   └── Grep Search
   ↓
验证与回退层
   ↓
日志与历史记录层
```

### 3.2 模块职责
| 模块 | 职责 |
|---|---|
| `core/` | 定义任务请求、任务状态、通用模型 |
| `planner/` | 负责 plan-execute 流程控制、步骤生成、等待用户确认 |
| `tools/` | 抽象七类核心工具能力与约束 |
| `executor/` | 调度工具执行具体步骤 |
| `safety/` | 风险识别、白名单、边界控制、确认机制 |
| `storage/` | 任务记录、快照、回退 |
| `ui/` | 展示任务输入、计划、日志、Diff 和结果 |

## 4. 为什么采用 Plan-Execute 模式

### 4.1 问题背景
如果用户一提交需求，Agent 就立刻进入执行，会有三个明显问题：
1. 用户不知道它准备改哪些文件
2. 用户不知道它准备运行哪些命令
3. 用户无法在高风险动作前进行人工确认

这会直接导致系统不可控，也不符合课程项目中“可解释、可验证、安全”的要求。

### 4.2 Plan-Execute 模式定义
本项目采用两阶段交互：

#### Plan 阶段
Agent 只做以下事情：
- 解析需求
- 扫描仓库与关键文件
- 生成步骤化执行计划
- 标识将调用的工具能力
- 列出待修改文件、待执行命令、风险标签
- 停止在“等待用户确认”状态

#### Execute 阶段
只有在以下条件满足后才进入：
- 用户确认计划
- 风险操作已被接受
- 执行边界（目录、命令）合法

然后系统才真正执行修改、运行命令、采集结果并输出 Diff。

### 4.3 选择 Plan-Execute 的理由
| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| 直接执行 | 开发快 | 风险高，不可解释，不可控 | 不选 |
| 纯建议模式 | 安全 | 不能形成 Agent 闭环 | 不选 |
| Plan-Execute | 安全、可解释、可交互、易测试 | 流程略长 | 选用 |

### 4.4 用户交互要求
在课程原型中，Plan 模式页面或面板至少应展示：
- 任务摘要
- 拆解步骤列表
- 待读取/修改的文件
- 待执行的 shell 命令
- 风险标签
- “确认执行 / 返回修改需求 / 取消” 按钮

这也是后续原型设计和测试用例的依据。

## 5. 工具层详细技术路线

### 5.1 Code Interpreter（代码解释器）
#### 目标
在安全边界内执行 Python 代码片段，用于：
- 数据清洗
- 结构化分析
- 小型验证脚本
- 结果计算与格式转换

#### 实现建议
- 使用独立 Python 进程执行
- 约束执行时长
- 限制输出大小
- 后续可升级为容器/沙盒

#### 不直接开放的能力
- 任意联网
- 任意系统写入
- 长时间后台运行

### 5.2 Bash Shell（命令执行）
#### 目标
执行如 `pytest`、`ruff check`、`pylint`、`npm test` 等命令。

#### 实现建议
- 基于 `subprocess.run()` 封装
- 必须指定工作目录
- 支持超时、退出码、stdout/stderr 捕获
- 对高风险命令（如 `rm -rf`、`git push --force`）做确认拦截

### 5.3 文件读取 / 重写 / 局部编辑
#### 设计拆分原因
这三者必须分开设计，原因如下：
- **读文件**：只读能力，用于上下文理解
- **写文件**：适合新建文件和整文件覆盖
- **编辑文件**：适合定点修改，避免误改整文件

#### 稳健路线
优先顺序：
1. 读文件建立上下文
2. 用 grep/glob 精确定位目标
3. 生成编辑计划
4. 用 edit 工具做局部修改
5. 必要时才使用 write 工具整文件重写
6. 强制输出 diff 验证修改效果

### 5.4 Glob / Grep 搜索
#### Glob 搜索主要解决
- 找入口文件
- 找测试文件
- 找配置文件
- 找特定后缀文件

#### Grep 搜索主要解决
- 找函数定义
- 找类名 / 配置键 / 报错字符串
- 找已有测试样例
- 找 TODO/FIXME 与调用链

#### 组合价值
Glob 解决“去哪里找”，Grep 解决“找什么内容”。二者组合是 Agent 高效理解仓库的关键。

## 6. 开发方法：TDD 优先，不后补测试

### 6.1 为什么本项目必须用 TDD
Coding Agent 相关功能具有明显的分支逻辑和高风险路径，例如：
- plan 模式是否停在确认阶段
- 高风险命令是否被拦截
- 编辑工具是否只改指定区域
- 越权路径是否被拒绝

这些逻辑如果先写代码、后补测试，极易出现“代码看似完成，行为却没有被证明”的问题。因此本项目采用 **TDD（测试驱动开发）** 作为实现原则。

### 6.2 TDD 执行流程
1. 先写失败测试
2. 运行测试，确认失败原因正确
3. 写最小实现使测试通过
4. 再运行测试确认通过
5. 重构并保持测试全绿
6. 最后同步更新文档

### 6.3 本项目中的 TDD 落地对象
- `validate_task_request()`：输入校验
- `PlanExecutionController`：plan / auto 分流
- `ToolCapability`：工具能力配置校验
- 后续的 shell 执行器、文件边界控制器、回退逻辑等

## 7. 质量保障与测试路线

### 7.1 静态质量门禁
- **Ruff**：发现风格问题、未使用导入、格式与常见错误
- **Pylint**：识别复杂度、命名、设计坏味道

### 7.2 自动化测试层次
| 测试类型 | 目标 |
|---|---|
| 单元测试 | 验证输入校验、风险识别、流程控制等核心函数 |
| 黑盒测试 | 从用户角度验证任务输入、模式切换、风险反馈 |
| 白盒测试 | 覆盖条件分支、异常路径、工具状态判断 |
| 覆盖率测试 | 衡量核心模块语句和分支覆盖 |

### 7.3 验证顺序
```bash
ruff check .
pylint src/codepilot
pytest --cov=src/codepilot --cov-report=term-missing
```

## 8. 与课程实验交付的对应关系
本技术路线能够直接支撑实验要求：
- **项目概述**：明确 Agent 定位与价值
- **用户故事**：围绕 plan-execute 主流程和工具能力展开
- **原型设计**：Plan 页、执行页、结果页都有明确结构
- **迭代计划**：第一轮做核心工具链与 plan-execute，第二轮做日志、边界控制、回退
- **测试方案**：既有静态检查，也有黑盒/白盒/覆盖率设计

---

## 9. 最终技术路线结论
本项目最终采用如下路线：
1. **以 Python 作为 Coding Agent 核心实现语言**
2. **以七类核心工具能力构建 Agent 执行闭环**
3. **以 Plan-Execute 模式保证过程可控、可解释、可交互**
4. **以 TDD 原则驱动核心模块实现，避免先写后补测试**
5. **以 Ruff + Pylint + Pytest + Coverage 建立质量门禁**
6. **以模块化设计为后续扩展沙盒、容器执行、历史回退和 Web 原型留出空间**

相比原先过于概括的方案，这一路线更贴近真正的 Coding Agent 工程问题：它不仅说明“用什么技术”，也说明“为什么这样分层、如何控制风险、怎样测试验证、如何与用户交互”。这才是一个可实施的技术路线。
