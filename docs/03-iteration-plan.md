# 两轮迭代计划

## 1. 迭代策略
项目采用两轮迭代：
- **迭代 1：第 1-6 周开发 + 第 7-8 周核心优化准备**
- **迭代 2：第 7-8 周增强与收尾**

原则是先完成 Coding Agent 的主流程，再补日志、失败处理、历史与回退等增强能力。

## 2. 用户故事优先级
### 高优先级（核心闭环）
- US-01 自然语言任务输入
- US-02 仓库扫描与上下文理解
- US-03 任务规划展示
- US-04 多文件代码修改
- US-05 命令执行与测试验证
- US-06 代码变更对比
- US-07 危险操作确认

### 中优先级（增强能力）
- US-08 执行日志记录
- US-09 失败修复与重试
- US-10 历史会话与任务记录
- US-11 执行边界控制
- US-12 一键回退

## 3. 迭代 1 计划（第 1-6 周）
### 目标
实现可演示的 Coding Agent 核心闭环：
`输入任务 → 理解仓库 → 输出计划 → 用户确认 → 执行修改 → 运行验证 → 展示 Diff`

该阶段必须同时落地三项约束：
1. 七类核心工具能力的抽象设计
2. Plan-Execute 交互主流程
3. 以 TDD 原则推动核心模块实现

- US-01

- US-03

- US-05

### 周计划
| 周次 | 主要任务 | 预期结果 |
|---|---|---|
| 第1周 | 需求分析、用户故事整理、原型草图 | Backlog 与原型框架 |
| 第2周 | 任务输入界面与会话流转、定义 plan 模式交互 | 可提交任务并进入分析 |
| 第3周 | 仓库扫描、目录树与关键文件识别、补充 glob/grep 设计 | 输出仓库摘要 |
| 第4周 | 任务规划、工具能力标注与执行前确认 | 展示计划并支持确认/取消 |
| 第5周 | 多文件修改、Diff 预览、按 TDD 实现关键骨架 | 支持核心文件编辑 |
| 第6周 | 命令执行、Python 解释执行、测试联调 | 核心流程贯通 |

### 迭代 1 验收标准
- 能输入自然语言任务
- 能展示仓库结构与关键文件
- 能给出步骤化计划
- 能修改至少 1~2 个文件
- 能运行至少一种测试或构建命令
- 能展示 Diff 与风险确认

## 5. 里程碑与产出
| 里程碑 | 时间 | 产出 |
|---|---|---|
| M1 需求冻结 | 第1周末 | 用户故事、优先级、原型草图 |
| M2 核心能力打通 | 第4周末 | 任务输入 + 扫描 + 规划 |
| M3 核心闭环完成 | 第6周末 | 修改 + 执行 + Diff 演示 |
| M4 增强能力完成 | 第8周末 | 日志 + 回退 + 历史 + 边界控制 |

## 6. 迭代管理建议
- 看板状态建议：`新建 / 进行中 / 已完成`
- 每个用户故事必须绑定负责人、故事点、优先级
- 每轮迭代结束后保留演示截图、测试记录与问题清单
- 对高风险功能（删除、覆盖配置、批量命令）单独标识并优先测试

## 7. 结论
两轮迭代的设计遵循“先核心、后增强”的原则，既满足实验手册对时间划分与 Sprint 组织的要求，也能保证项目在有限周期内完成可运行原型与可验证交付。
